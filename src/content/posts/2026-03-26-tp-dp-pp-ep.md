很多人第一次听 TP / DP / PP / EP，会有一种熟悉的迷糊感：

- 每个词都见过
- 每个词都知道“和并行有关”
- 但一旦放到同一个训练集群里，就不知道它们到底在切哪一刀

这也是为什么很多关于并行策略的讨论会越说越乱。  
因为如果没有一个统一坐标系，TP、DP、PP、EP 很容易被讲成四个互相竞争的选项；但实际情况正好相反，它们往往是 **叠加关系**。

这篇文章想做的事很简单：  
把这四种并行方式放回同一张地图里，先讲它们分别切什么，再讲它们为什么常常要混着用。

## 先记住一句最省事的话

如果只想先记一个版本，可以先记这句：

- **DP**：切 batch，模型复制多份
- **TP**：切单层张量，一层拆到多卡
- **PP**：切模型深度，不同层放不同卡
- **EP**：切 MoE experts，把不同专家放不同卡

这四种并行的差异，本质上不在名字，而在 **它们切分的对象不同**。

## DP：最自然，也最先碰到显存上限

数据并行是最直觉的一种并行方式。  
官方文档的定义也最朴素：**把模型复制到多张 GPU 上，再把 batch 切开。**

这样每张卡都执行相同的前向和反向，只是吃到的数据不同。  
训练一步结束后，再通过梯度同步把各个副本拉齐。

这也是为什么最传统的 DDP，本质上就是：

- 每卡一份完整模型
- 每卡一份完整优化器状态
- 每卡处理自己的 mini-batch shard
- 训练步末尾用 all-reduce 同步梯度

它的好处非常明显：实现最直接、吞吐扩展也通常最好。  
因为每张卡算的都是完整模型，计算图最干净，很多时候性能也最稳定。

但它的问题同样直接：

**模型太大时，单卡根本放不下完整参数和优化器状态。**

这也是为什么 DeepSpeed 的 ZeRO 会被经常放进“DP”这个语境里讨论。  
ZeRO 并没有改变“数据并行”的基本思想，它做的是对 DP 的内存冗余做优化：

- ZeRO-1：分 optimizer states
- ZeRO-2：再分 gradients
- ZeRO-3：连 parameters 也分

所以一个容易记错但必须讲清楚的点是：

**ZeRO 不是第五种并行。它更像 DP 的省显存版本。**

## TP：当一层都塞不进单卡时，就得横着切

TP 最适合解决的问题不是“吞吐不够”，而是：

**单层太大，单卡内存扛不住。**

Megatron 官方文档对 TP 的定义很清楚：  
它把单个层的参数张量拆到多张 GPU 上。

换句话说，TP 不是把“前 20 层放一张卡、后 20 层放另一张卡”，那是 PP；  
TP 做的是把同一层里的矩阵运算拆开，让几张卡一起完成这一层。

它带来的直接收益有两个：

1. 单卡模型状态更小
2. 某些中间张量和激活也会随之缩小

但代价也很直白：

**层内通信会变多。**

因为原来在单卡里做完的一层，现在变成了“多卡共同完成的一层”，中间结果得靠 collective 去同步。

这也是为什么 TP 虽然很强，但并不是“越大越好”。  
如果把 TP 拉得太高，通信会开始压过算力收益，尤其是跨节点场景。

可以把 TP 理解成一句非常工程的话：

**它优先解决“层太宽”这个问题。**

所以 TP 通常更适合放在高带宽、低时延的 GPU 域内，比如同机 NVLink / NVSwitch，而不是轻易跨机乱拉。

## PP：当模型整体太深时，就得按层往下切

如果说 TP 是横着切一层，那 PP 就是竖着切整个模型。

官方文档的定义是：**把连续层分配到不同 GPU 上。**

例如：

- 0-19 层在 GPU 0
- 20-39 层在 GPU 1
- 40-59 层在 GPU 2
- 60-79 层在 GPU 3

这样做的最大价值不是减少单层大小，而是：

**把整个模型深度分摊到更多 GPU。**

PP 的问题也非常经典：pipeline bubble。  
因为各 stage 不会在同一时刻都满负荷工作，前后总会有“灌流水线”和“排空流水线”的空洞。

这也是为什么 PP 的关键不只是“切层”，而是 **怎么排程 micro-batches**。  
Megatron-LM 论文和官方文档里都强调了 interleaved pipeline schedule 的价值：把每个 pipeline stage 再切成多个 model chunks，用更细粒度的调度去压 bubble。

所以 PP 真正解决的问题可以概括成一句话：

**当模型整体太深、单机装不下时，用层间切分把它铺开。**

## EP：只有到了 MoE，expert 才值得单独切

EP 是最容易被误解的一种并行。

因为它看起来像“另一种模型并行”，但官方文档专门强调了一点：

**EP 只作用于 MoE expert 层，不会改变其他普通层的并行映射。**

这句话非常关键。

如果模型是稠密模型，没有 experts，那根本谈不上 EP。  
EP 的存在前提，是模型里有一组可以被 router 动态激活的 experts。

这时 EP 做的事情是：

- 把不同 experts 分配到不同 GPU
- token 先经过 router
- 再把 token 分发给对应 expert 所在的 GPU

所以 EP 解决的问题不是“单层太宽”，也不是“整模型太深”，而是：

**expert 太多、每个 token 又只会激活其中少数几个，适合把 experts 分散到不同 GPU。**

但 EP 的复杂度也最高，因为除了参数切分，还会引入：

- token dispatch
- all-to-all 通信
- router 负载均衡
- expert 热点和长尾问题

这也是为什么很多人第一次接触 EP 时会觉得它“不像前面三种那么干净”。  
确实如此。因为 EP 面对的不是一个规则的稠密矩阵，而是一套动态路由系统。

## 为什么真实系统里常常是混合并行

到这里就能看出，TP / DP / PP / EP 并不是四种互斥选项。  
它们分别解决的是不同瓶颈：

- DP 解决吞吐扩展
- TP 解决单层太大
- PP 解决整模型太深
- EP 解决 MoE expert 太多

所以真实大模型训练里，更常见的不是“选一个”，而是把它们拼起来。

Megatron Core 文档甚至直接把公式写出来了：

`Total GPUs = TP × PP × CP × EP × DP`

如果这篇先只看 TP / DP / PP / EP，那可以临时把 CP 放一边，先记成：

`World Size ≈ TP × PP × EP × DP`

## 一套更实用的判断顺序

很多时候，真正的问题不是“这四个怎么定义”，而是：

**什么时候该先加哪一个？**

如果按工程判断顺序，通常可以这么想：

### 1. 先看是不是能只用 DP

如果模型还能放进单卡，先用 DP 往往是最省脑子的。

因为 DP：

- 实现最成熟
- 调试最容易
- 吞吐扩展通常最好

只有当模型已经大到单卡放不下，或者优化器状态显存爆掉，才需要继续加别的维度。

### 2. 模型放不下时，先想 TP 还是 PP

这一步常见的误区，是把 TP 和 PP 都当成“模型并行”然后随便选。

更好的问法应该是：

- 问题是“单层太宽”吗？那优先想 TP
- 问题是“整模型层数太深/整体装不下”吗？那优先想 PP

很多超大模型最后会 TP + PP 一起上，就是因为这两种问题经常同时存在。

### 3. 只有 MoE 才谈 EP

EP 不是通用平替。  
如果模型本身不是 MoE，EP 根本不在候选集里。

一旦进入 MoE 语境，才需要额外考虑：

- 专家数和 expert parallel size 是否整除
- token dispatch 的网络代价
- router 是否会导致热点专家
- 是否需要 expert tensor parallelism

## 最容易记混的几个点

最后把几个最容易混淆的点单独挑出来说。

### TP 和 PP 都属于 model parallel，但切分维度不同

这一点经常被一句“模型并行”给糊过去。  
但工程上差别非常大：

- TP：一层内部切
- PP：层与层之间切

这决定了它们的通信模式、适用拓扑和性能瓶颈都不一样。

### ZeRO 不是 TP / PP / EP 的替代品

ZeRO 处理的是 DP 的内存冗余问题。  
它不会把一层拆到多卡，也不会把层深切开，更不会处理 MoE experts。

所以“用了 ZeRO 还要不要 TP / PP”这个问题本身就问偏了。  
答案通常是：**要看模型是否已经大到仅靠优化版 DP 还不够。**

### EP 不是“把所有层都专家化”

EP 只发生在 expert 层。  
其他普通层依然可能走 TP / PP / DP 的组合。

所以在一个混合并行系统里，很常见的现实是：

- 稠密部分按 TP + PP + DP 跑
- MoE 部分再额外叠 EP

## 最后一句话

如果只记一句话，我会这样概括：

**TP / DP / PP / EP 的区别，不在名字，而在“到底切的是 batch、层内张量、模型深度，还是 MoE experts”。**

一旦把这个坐标系记住，后面的很多配置就会顺很多。

因为真正的并行策略设计，从来不是“选一个名词”，而是按瓶颈来切：

- 吞吐不够，先想 DP
- 层太大，补 TP
- 模型太深，补 PP
- 模型是 MoE，再考虑 EP

这才是把四个词讲清楚之后，真正能拿去用的部分。
